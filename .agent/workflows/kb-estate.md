# KB부동산 아파트 단지 데이터 수집 시스템 요구사항 v1.0
(목표: “수집 + 운영/감사/모니터링/재현성”까지 포함한 전문 수집 시스템)

---

## 1) 용어/도메인 정의

### 1.1 수집 대상(요청 항목)
- 대상 플랫폼: KB부동산(아파트 단지)
- 단지별/면적별 수집 항목
  1) KB 시세(매매): 일반가, 상위평균가, 하위평균가, 기준일
  2) 실거래가(매매): 거래일, 거래금액, 면적, 층(가능하면), 신고일(가능하면)
  3) 매매 호가(매물): 등록일/업데이트일, 호가, 면적, 층(가능하면), 상태(거래가능/거래완료 등)

### 1.2 엔터티(핵심 테이블 개념)
- Complex(단지)
- Area(단지 내 면적 라인업: 전용/공급/평형/면적코드)
- KBPrice(시세 스냅샷 + 시계열)
- Transaction(실거래 레코드)
- Listing(매물/호가 레코드)
- CrawlJob/CrawlRun/CrawlTask(수집 작업/실행/태스크)
- RawPayload(원문 스냅샷: 재현/감사/디버깅용)

## 2) 시스템 목표/비목표

### 2.1 목표
- “단지/면적별” 가격 데이터를 정규화해서 저장하고, 히스토리를 유지한다.
- 실패/차단/구조 변경에 강한 수집 파이프라인을 구축한다.
- 운영자가 수집을 “통제/관찰/재실행/감사”할 수 있는 관리자 화면 제공.
- 데이터 품질(누락/이상치/중복)을 자동 감지.

### 2.2 비목표(초기 범위 제외)
- 개인 연락처/중개사 개인정보 수집 및 저장
- 사용자별 개인화 추천/알림(2차 단계)
- 실시간(초 단위) 호가 변동 추적(대량 비용/리스크 큼)

## 3) 컴플라이언스/보안 요건 (필수)

### 3.1 준수 체크리스트
- [ ] 타겟 데이터의 이용약관/라이선스 검토 완료(법무/내부정책)
- [ ] robots.txt/접근정책 확인(가능한 경우)
- [ ] 대량 수집 시 “사전 허가/제휴” 확보
- [ ] 개인정보/연락처/민감정보 저장 금지(필요 시 마스킹/해시)
- [ ] 계정/쿠키/토큰은 Secret Manager에 저장(코드/로그에 금지)
- [ ] RawPayload에 개인정보가 섞일 위험이 있으면 저장 전 필터링/마스킹

### 3.2 기술 보안
- 수집 워커는 아웃바운드 도메인 allowlist 적용
- 관리자 화면 RBAC(권한 분리: Admin/Operator/Viewer)
- 감사로그(Audit log): 누가 어떤 단지/기간을 수집/삭제/재처리했는지 기록

## 4) 수집 파이프라인 요건(전문 수집 시스템 핵심)

### 4.1 수집 방식(커넥터 전략)
- Connector 인터페이스를 표준화하고, 소스별 구현체를 “플러그인”처럼 추가
- 권장 커넥터 구성
  - KBPriceConnector (KB 시세)
  - KBListingConnector (매물/호가)
  - MolitTransactionConnector (실거래가: 공공 OpenAPI)
- KB 계열은 “허가된 접근 방식” 기준으로 구현(공식 제공 채널/제휴 API/계약된 데이터 공급처 등)

### 4.2 스케줄/증분 수집
- 기본 정책(권장)
  - KB 시세: 1일 1회(또는 주 2~3회) 증분 갱신 + 변경 감지 시에만 저장
  - 실거래가: 1일 1회(신고 지연/정정 반영 위해 최근 3개월 롤링 재수집)
  - 매물/호가: 1일 1~2회(비용/차단 리스크 고려) + 단지별 우선순위 큐
- 증분 기준키
  - 시세: (complex_id, area_id, 기준일)
  - 실거래: (complex_id, area, 계약일, 금액, 층) 조합 unique
  - 매물: (source_listing_id) unique

### 4.3 실패/재시도/회복탄력성
- 재시도 전략: exponential backoff + jitter
- 오류 분류:
  - 네트워크/타임아웃(재시도)
  - 인증 만료(토큰 재발급/세션 갱신)
  - 구조 변경(파서 실패 → 알림 + RawPayload 저장)
  - 차단/429(속도 낮춤, 스케줄 조정, 운영자 알림)
- “아이템 단위” 재수집 가능(특정 단지/면적만 재실행)

### 4.4 데이터 품질(QA)
- 저장 전 Validation Rules 예시
  - 가격이 음수/0이면 reject
  - 전일 대비 변동률이 임계치 초과하면 “이상치 플래그”
  - 동일 키 중복 발생 시 dedup
  - 필수 필드 누락률이 임계치 초과하면 RUN을 “경고” 상태로 마킹
- 품질 리포트 자동 생성(일/주 단위)

### 4.5 원문(Raw) 보관/재현성
- 각 수집 결과는 RawPayload(원문 JSON/HTML)와 함께 보관(선택)
- 저장 방식:
  - DB에는 메타만 저장 + 실제 원문은 Object Storage(S3/MinIO)에 저장 권장
- 원문에는 PII 필터를 적용(필요 시)

## 5) 관리자 화면(운영 콘솔) 요구사항

### 5.1 메뉴 구조(권장 IA)
1) 대시보드
2) 단지 관리
3) 수집 작업(Jobs)
4) 실행 이력(Runs) / 태스크(Task) 로그
5) 데이터 탐색기(시세/실거래/호가 조회)
6) 품질 리포트
7) 알림/규칙 설정
8) 시스템 설정(커넥터/크레덴셜/레이트리밋)
9) 사용자/권한(RBAC)
10) 감사 로그(Audit)

### 5.2 핵심 화면별 기능

#### (1) 대시보드
- 오늘/최근 24h 수집 성공률, 실패율, 평균 소요시간
- 오류 Top N(파서 실패/인증 실패/차단)
- 큐 적체(대기 태스크 수), 워커 헬스(heartbeat)
- “이상치 탐지” 알림 카드(급등락, 누락률 증가)

#### (2) 단지 관리
- 단지 등록/수정/비활성화
- 단지 우선순위(High/Normal/Low)
- 수집 범위 설정:
  - 면적 전체 vs 특정 면적만
  - 매물 수집 on/off
- 단지 상세 탭:
  - 기본정보
  - KB시세(시계열/스냅샷)
  - 실거래(리스트/필터)
  - 호가(리스트/통계: 최저/최고/중앙/평균/건수)
  - 수집 이력(최근 RUN 링크)

#### (3) 수집 작업(Jobs)
- Job 정의:
  - Job 종류(KB시세/실거래/호가)
  - 대상(전체/지역/단지그룹/단지 1개)
  - 스케줄(cron)
  - 동시성/레이트 제한
- “지금 실행(Run Now)”, “중지/재개”, “실패 태스크 재시도”

#### (4) 실행 이력(Runs) & 로그
- Run 단위 상세:
  - 시작/종료, 총 처리 건수, 성공/실패, 에러 샘플
  - 태스크 단위 드릴다운
  - RawPayload 링크(권한 있는 사용자만)
- 검색: 단지명/기간/에러코드/커넥터

#### (5) 데이터 탐색기
- 단지/면적/기간 필터
- CSV Export
- 시세/실거래/호가 “원천 소스/수집시각” 표시(데이터 계보)

#### (6) 품질 리포트
- 누락률, 중복률, 이상치 건수, 파서 실패율 트렌드
- 룰별 발생 현황 + 샘플 데이터 링크

#### (7) 알림/규칙
- 알림 채널: Email/Slack/Webhook
- 트리거:
  - 연속 N회 실패
  - 파서 실패율 급증(사이트 구조 변경 추정)
  - 429/차단 감지
  - 특정 단지 가격 급등락(임계치)

## 6) API/서비스 아키텍처(권장)

### 6.1 구성요소
- Admin Web (React/Next.js 등)
- Admin API (FastAPI/NestJS)
- Scheduler (Celery Beat / Airflow 중 택1)
- Worker Pool (Celery/RQ/Sidekiq 등)
- DB (PostgreSQL)
- Cache/Queue (Redis)
- Object Storage (S3/MinIO) : RawPayload 보관
- Observability
  - Logs: 구조화 로그(JSON)
  - Metrics: Prometheus
  - Tracing: OpenTelemetry(선택)
  - Error: Sentry(선택)

### 6.2 데이터 흐름
[Admin] 단지/스케줄 설정
   ↓
[Scheduler] Job 생성
   ↓
[Queue] Task 적재(단지/면적 단위)
   ↓
[Workers] Connector 실행 → Normalize/Validate → Store(DB) + Raw(Object storage)
   ↓
[Admin] Runs/데이터/품질 확인 & 재실행

## 7) 수집 데이터 스펙(정규화 기준)

### 7.1 KB 시세(매매) 스냅샷(면적별)
- 기준일(as_of_date)
- 일반가(general_price)
- 상위평균가(high_avg_price)
- 하위평균가(low_avg_price)
- 통화단위: KRW (정수, 원 단위로 정규화)
- 원천 메타: source, fetched_at, payload_hash, parser_version

### 7.2 실거래가(매매)
- 계약일(contract_date)
- 거래금액(price)
- 전용면적(exclusive_m2)
- 층(floor, optional)
- 신고일(reported_date, optional)
- 정정/해제 여부(optional)

### 7.3 매매 호가(매물)
- listing_id(원천 고유키)
- 호가(ask_price)
- 면적(exclusive_m2 or area_id)
- 층(floor, optional)
- 상태(status: active/sold/removed/unknown)
- 등록/갱신시각(posted_at/updated_at, optional)
- 개인정보/연락처 저장 금지

---

## 8) 운영 품질 기준(SLO 권장)
- 일간 수집 성공률 ≥ 98%
- 파서 실패율 ≤ 1% (초과 시 구조 변경 알림)
- 데이터 누락률(필수필드) ≤ 0.5%
- 백필(최근 3개월 실거래) 일 1회 완료
