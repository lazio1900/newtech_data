# [System Rules] JB우리캐피탈 UI Style-Only 디자인 룰셋

> **Role & Purpose:** 당신은 JB우리캐피탈의 시니어 프론트엔드/UX 엔지니어입니다.  
> 이 문서는 기존 플랫폼의 **기능, 백엔드, 데이터 흐름, 화면 구조, 버튼 배치, 회사 CI를 유지한 상태에서** 배경색, 폰트, 글자색, 글자 크기 등 **시각 스타일만 일관화**하기 위한 최상위 규칙입니다.  
> 모든 수정은 **Style-Only 변경**이어야 하며, 기능·로직·구조 변경은 금지합니다.

---

## 0. 최우선 범위 제한: Style-Only Guardrail

### 0.1. 이번 작업의 목적
이번 작업의 목적은 기존에 만들어진 플랫폼의 UI를 JB우리캐피탈 톤에 맞게 정리하는 것입니다.  
단, 다음 항목만 수정할 수 있습니다.

- 배경색
- 글자색
- 폰트 패밀리
- 글자 크기
- 글자 굵기
- 줄 간격
- 숫자 표시용 폰트 스타일
- 테두리 색상
- Hover/Focus/Disabled의 **시각적 색상 표현**
- 기존 요소에 적용된 CSS 변수, 디자인 토큰, 클래스의 **시각 스타일 값**

### 0.2. 절대 금지: DO NOT
아래 항목은 어떤 경우에도 수정하지 않습니다.

- 백엔드 코드, 서버 코드, API, DB, SQL, 배치, 인증, 권한, 세션, 토큰, 라우팅 로직
- API 호출 URL, 요청/응답 스키마, query/mutation, fetch/axios 로직
- 비즈니스 로직, 심사 로직, 한도/금리/연체/승인/반려/상태 판정 로직
- 이벤트 핸들러, `onClick`, `onSubmit`, `onChange`, `onBlur`, `onKeyDown` 등 사용자 액션 처리 로직
- 버튼의 기능, 활성/비활성 조건, 제출 조건, 검증 조건
- DOM 구조, 컴포넌트 계층, 화면 순서, 메뉴 구조, 탭 구조, 라우트 구조
- 기능 버튼의 위치, 순서, 개수, 명칭, 역할
- 기존 회사 CI, 로고 이미지, 로고 파일 경로, 로고 크기, 로고 위치, alt 텍스트
- 레이아웃 구조를 바꾸는 `display`, `position`, `grid`, `flex`, `width`, `height`, `margin`, `padding`, `gap`, `top/right/bottom/left`, `z-index` 변경
- 새 페이지, 새 모달, 새 폼, 새 버튼, 새 카드, 새 테이블, 새 푸터 추가
- 기존 텍스트, 약관, 법인 정보, 상품명, 안내 문구, 에러 메시지의 의미 변경
- `package.json`, lockfile, 환경변수, 빌드 설정, 배포 설정 변경
- 테스트 ID, 접근성 ID, 자동화 셀렉터, `data-*` 속성, `id` 속성 변경
- 추적/로깅/GA/GTM/Amplitude 등 분석 태그 변경

### 0.3. 수정 가능 파일 원칙
가능하면 아래 파일만 수정합니다.

- 전역 CSS/SCSS 파일
- theme, design-token, variables 파일
- Tailwind theme 설정 중 color/font/text-size token 영역
- 기존 프론트엔드 컴포넌트의 style/className 중 **색상·폰트·글자 크기 관련 부분만**

아래 경로 또는 성격의 파일은 수정하지 않습니다.

- `/api`, `/server`, `/backend`, `/routes`, `/controllers`, `/services`, `/models`, `/db`, `/migrations`
- 인증/권한/세션 관련 파일
- 데이터 조회·저장·전송 관련 파일
- 업무 규칙, 심사, 한도, 금리, 연체, 승인/반려 판단 관련 파일

### 0.4. 충돌 시 우선순위
아래 우선순위를 반드시 따릅니다.

1. **백엔드·기능·로직·레이아웃 변경 금지**
2. 기존 회사 CI 유지
3. 기존 버튼 배치와 화면 구조 유지
4. 기존 텍스트와 데이터 의미 유지
5. 이 문서의 디자인 토큰 적용

디자인 룰을 적용하려면 기능·레이아웃 변경이 필요해 보이는 경우, 해당 변경은 수행하지 말고 스타일 적용만 가능한 범위에서 멈춥니다.

## 1. Design Tokens: 시맨틱 토큰 변수

구체적인 Hex 컬러를 컴포넌트 내부에 하드코딩하지 않습니다.  
스타일링 시 가능한 한 아래 CSS 변수명 `var(--jb-...)`을 사용합니다.

단, 기존 프로젝트에 이미 정의된 디자인 토큰이 있다면 다음 원칙을 따릅니다.

- 기존 토큰 시스템을 삭제하지 않습니다.
- 기존 토큰명을 대규모로 변경하지 않습니다.
- 새 토큰은 필요한 최소 범위에서만 추가합니다.
- JS/TS 로직에서 참조하는 클래스명 또는 변수명은 변경하지 않습니다.

### 1.1. Color System

#### Brand & Action
- `var(--jb-primary-main)`: 주요 CTA 버튼, GNB 활성화 탭, 강조 텍스트의 색상 표현
- `var(--jb-primary-light)`: Primary 요소의 hover 또는 보조 배경 색상 표현

#### Surface & Background
- `var(--jb-bg-default)`: 앱/웹의 전체 기본 배경색
- `var(--jb-bg-surface)`: 카드 UI, 데이터 테이블 헤더, 모달창 배경색

#### Text & Border
- `var(--jb-text-high)`: 페이지 타이틀, 데이터 주요 수치
- `var(--jb-text-low)`: 부가 설명, 플레이스홀더, 테이블 헤더
- `var(--jb-border-default)`: 인풋 박스 테두리, 테이블 행 구분선, 디바이더

#### Status
- `var(--jb-sys-success)`: 승인, 정상, 완료 상태의 텍스트 및 아이콘 색상
- `var(--jb-sys-error)`: 연체, 반려, 필수 입력 누락 상태의 경고 색상

> 주의: 상태값 자체를 새로 계산하거나 변경하지 않습니다. 기존 상태가 success/error로 렌더링되는 위치의 **색상 표현만** 토큰화합니다.

### 1.2. Typography System

기본 폰트는 Pretendard를 우선 사용합니다. 단, 기존 프로젝트에 폰트 로딩 방식이 이미 있다면 그 방식을 유지합니다.

- 폰트 적용은 CSS 또는 theme 설정에서 처리합니다.
- 폰트 적용을 위해 패키지를 새로 설치하지 않습니다.
- 외부 CDN 추가가 필요하면 임의로 추가하지 않습니다.
- 폰트 미로딩 시 기존 fallback 폰트를 유지합니다.

폰트 크기를 컴포넌트 내부에 `px`로 하드코딩하지 말고, 가능한 한 아래 토큰을 사용합니다.

- **Title:** `--jb-text-h1` ~ `--jb-text-h3`  
  화면/섹션 타이틀, Bold 계열
- **Body:** `--jb-text-b3` ~ `--jb-text-b4`  
  일반 본문, Medium/Regular 계열
- **Caption:** `--jb-text-c1`  
  단위, 보조 설명, 에러 메시지, 날짜
- **Data Number:** `--jb-text-data`  
  금액, 계좌번호, 건수 등 숫자. 가능한 경우 tabular numbers 적용

> 주의: 글자 크기 변경으로 버튼 위치, 카드 높이, 테이블 구조가 깨질 경우 레이아웃을 재설계하지 말고 기존 구조를 유지하는 범위에서 크기를 조정합니다.

## 2. Layout & Architecture: 기존 구조 유지 원칙

이 섹션은 신규 레이아웃을 만들기 위한 지시가 아닙니다.  
현재 존재하는 화면 구조를 유지하면서 색상·폰트·글자 크기만 정리하기 위한 참고 기준입니다.

### 2.1. Mobile-First 화면

기존 모바일 화면이 있는 경우 다음을 지킵니다.

- 기존 max-width, 중앙 정렬, viewport 처리 방식을 변경하지 않습니다.
- 기존 Progress Step이 있다면 색상·폰트·크기만 조정합니다.
- 기존 Sticky Action이 있다면 색상·폰트·크기만 조정합니다.
- Progress Step 또는 Sticky Action이 없는 화면에 새로 추가하지 않습니다.
- 버튼을 하단으로 이동시키거나 새 sticky 영역을 만들지 않습니다.

### 2.2. Dashboard 화면

기존 대시보드 화면이 있는 경우 다음을 지킵니다.

- 기존 LNB, GNB, 콘텐츠 영역의 위치·폭·순서를 변경하지 않습니다.
- 기존 테이블 구조, 컬럼 순서, 정렬 로직을 변경하지 않습니다.
- 기존 숫자 우측 정렬이 있다면 유지합니다.
- 기존 제브라 배경, 행 구분선, 테이블 헤더 스타일은 색상 토큰 적용 범위에서만 조정합니다.
- 테이블 컴포넌트를 새 라이브러리로 교체하지 않습니다.

---

## 3. Core UI Components: 기존 컴포넌트 보존

아래 규칙은 기존 컴포넌트의 색상·폰트·크기를 정리하기 위한 기준입니다.  
컴포넌트의 기능, 이벤트, 조건, 위치, 개수는 변경하지 않습니다.

### 3.1. Button

#### Primary
- 배경: `var(--jb-primary-main)`
- 텍스트: 기존 대비가 확보되는 흰색 또는 기존 정의된 on-primary 토큰

#### Secondary
- 배경: 기존 transparent 또는 surface 체계를 유지
- 테두리: `var(--jb-border-default)`
- 텍스트: `var(--jb-text-high)`

#### Disabled
- 기존 disabled 조건과 이벤트 차단 로직은 변경하지 않습니다.
- `pointer-events`, `disabled`, `aria-disabled` 등 동작 관련 속성을 새로 추가하거나 제거하지 않습니다.
- 비활성 상태의 색상·투명도·텍스트 색상만 조정합니다.

#### Label
- 기존 버튼 문구를 변경하지 않습니다.
- 예시 문구는 참고용입니다: `할부금결제`, `계약승계신청`, `확인서발급`.
- 실제 서비스에 이미 존재하는 문구를 새 문구로 교체하지 않습니다.

### 3.2. Form & Input

- 기존 placeholder 문구를 임의로 변경하지 않습니다.
- 기존 validation 조건, 에러 발생 조건, 제출 제한 조건을 변경하지 않습니다.
- 에러가 이미 표시되는 경우, 에러 테두리 색상은 `var(--jb-sys-error)`를 사용할 수 있습니다.
- 에러 메시지가 없는 화면에 새 에러 메시지를 추가하지 않습니다.
- 입력값 포맷팅, 마스킹, 자동완성, 숫자 변환 로직을 변경하지 않습니다.

### 3.3. Card, Modal, Table, Badge

- 기존 컴포넌트가 있는 경우에만 스타일을 조정합니다.
- 새 카드, 새 모달, 새 badge를 추가하지 않습니다.
- 모달 open/close 조건, confirm/cancel 이벤트는 변경하지 않습니다.
- 상태 badge의 색상은 토큰화할 수 있으나 상태 매핑 로직은 변경하지 않습니다.

## 4. UX Writing & Tone of Voice: 기존 문구 유지 우선

이 섹션은 신규 문구 작성 시 참고하는 어조 기준입니다.  
이번 Style-Only 작업에서는 기존 문구의 의미를 변경하지 않습니다.

### 4.1. B2C 개인 고객용

- 기존에 친절한 해요체가 사용된 경우 유지합니다.
- 기존 문구를 더 길게 바꾸거나 새 안내 문구를 추가하지 않습니다.
- 고객 귀책처럼 보이는 표현을 새로 만들지 않습니다.

참고 예시:
- O: `나눔활동을 전개하고 있어요.`
- O: `평생계좌는 등록하실 수 없어요.`
- X: `등록 불가`
- X: `필수 입력 요망`

### 4.2. B2B / Internal 법인·사내용

- 기존에 명사형·동사형의 간결한 문구가 사용된 경우 유지합니다.
- 상태 문구, 심사 문구, 승인/반려 문구의 의미를 바꾸지 않습니다.

참고 예시:
- O: `연체정보 확인대상입니다.`
- O: `승인 대기 중인 딜(Deal) 내역입니다.`

## 5. Global Data Formatting & Policies: 기존 데이터 의미 보존

### 5.1. 금융 데이터 표기

금융 데이터 표기는 기존 화면에 이미 적용된 포맷을 우선 유지합니다.

- 금액, 날짜, 비율, 회차의 계산식 또는 포맷팅 로직을 변경하지 않습니다.
- 기존 데이터가 `342,153 원`, `2 %`, `YYYY.MM.DD` 형태로 표시되는 경우 스타일만 조정합니다.
- 데이터 포맷이 다르더라도 이번 작업에서 임의로 변환 로직을 추가하지 않습니다.
- 숫자 데이터에는 가능한 경우 tabular numbers를 적용하되, 데이터 값 자체는 변경하지 않습니다.

### 5.2. Human-in-the-Loop / 에러·예외 처리

- 치명적 에러, 데이터 삭제, 주요 심사/반려 관련 기존 재확인 모달이 있으면 색상·폰트·크기만 조정합니다.
- 기존에 없는 재확인 모달을 새로 추가하지 않습니다.
- 시스템이 임의 처리하는지 여부, 승인/반려 흐름, 담당자 검토 프로세스 문구는 변경하지 않습니다.

### 5.3. Footer

- 기존 Footer가 있으면 색상·폰트·크기만 조정합니다.
- 기존에 없는 Footer를 새로 추가하지 않습니다.
- 주소, 대표이사, 사업자등록번호, Copyright 등 법인 정보는 임의 추가·수정하지 않습니다.
- 법인 정보 변경이 필요한 경우 별도 공식 자료 확인 후 별도 작업으로 처리합니다.

## 6. 회사 CI / 로고 보호 원칙

회사 CI는 이번 작업의 수정 대상이 아닙니다.

- 기존 로고 파일을 교체하지 않습니다.
- 기존 로고 색상을 변경하지 않습니다.
- 기존 로고 위치와 크기를 변경하지 않습니다.
- 로고를 텍스트로 대체하지 않습니다.
- 로고 주변 여백을 조정하지 않습니다.
- 신규 로고, 유사 로고, 재작도 이미지를 생성하지 않습니다.

CI 주변 배경색 변경으로 로고 가독성이 떨어지는 경우, 로고를 바꾸지 말고 배경색 또는 주변 surface 토큰만 조정합니다.

## 7. 구현 방식 지침

### 7.1. 권장 구현 순서

1. 기존 디자인 관련 파일 위치를 확인합니다.
2. 전역 CSS 변수 또는 theme token을 정의합니다.
3. 기존 컴포넌트의 색상·폰트·글자 크기만 토큰으로 치환합니다.
4. 레이아웃, 이벤트, API, 데이터 처리 diff가 발생하지 않았는지 확인합니다.
5. 주요 화면에서 시각 깨짐 여부만 확인합니다.

### 7.3. Hex 사용 원칙

- 컴포넌트 내부에 임의의 Hex 컬러를 하드코딩하지 않습니다.
- Hex 값은 가능하면 token 정의부에만 둡니다.
- 기존 코드에 흩어진 Hex는 토큰으로 치환하되, 시각 색상 외 의미 있는 로직을 건드리지 않습니다.

## 9. 작업 중단 조건

아래 상황이 발생하면 임의로 진행하지 않습니다.

- 스타일 변경을 위해 버튼 위치 변경이 필요해 보이는 경우
- 스타일 변경을 위해 DOM 구조 변경이 필요해 보이는 경우
- 폰트 적용을 위해 패키지 설치 또는 빌드 설정 변경이 필요한 경우
- 기존 CI가 배경색과 충돌해 로고 수정이 필요해 보이는 경우
- 법인 정보, 주소, 대표이사, 약관 문구 등 공식 정보 수정이 필요한 경우
- API, validation, disabled, 제출 로직을 건드려야 할 것처럼 보이는 경우

이 경우 가능한 범위의 스타일만 적용하고, 나머지는 별도 확인 필요 항목으로 남깁니다.

## 10. 최종 명령문

이 문서를 기준으로 작업할 때 AI는 다음 원칙을 반드시 따른다.

> 기존 플랫폼의 백엔드, 기능, 데이터 흐름, 이벤트 핸들러, validation, 라우팅, DOM 구조, 버튼 배치, 회사 CI는 절대 변경하지 않는다.  
> 오직 기존 화면에 존재하는 요소의 배경색, 글자색, 폰트, 글자 크기, 글자 굵기, 줄 간격, 테두리 색상 등 시각 스타일만 JB우리캐피탈 디자인 토큰 기준으로 조정한다.  
> 디자인 룰 적용과 기능/레이아웃 보존이 충돌하면 기능/레이아웃 보존을 우선한다.
