## 2. 진단

24시간 내 실패 원인 분포:

```
ConnectorError  46건  ─ Server disconnected 34 + kb_area_code not found 12
IntegrityError  21건  ─ UNIQUE 인덱스 위반 (Transaction)
ZombieTimeout   19건  ─ worker SIGSEGV/OOM 후 RUNNING 상태로 남은 task
DataError        4건  ─ ask_price overflow
PendingRollbackError ─ facility UNIQUE 위반 (전체 run 차단)
```

근본 원인:
1. **schema** — `listings.ask_price` Integer 컬럼 (21억 이상 매물 overflow)
2. **schema** — UNIQUE 인덱스 부재로 동일 거래/시세/매물 중복 INSERT 누적 → 후일 인덱스 추가 시 한 번에 깨짐
3. **save logic** — `Transaction` 만 `db.merge()` 사용 (PK 없는 신규 객체에 merge 는 INSERT 와 동일, UPSERT 아님)
4. **save logic** — `ComplexFacility` 는 KB/OSM 응답에 동일 `external_id` 가 중복 포함되는 경우가 있는데 ON CONFLICT 미적용
5. **resilience** — `kb_listing` 외 connector 에 transient retry 미적용
6. **bookkeeping** — worker 비정상 종료 시 RUNNING/PENDING 으로 멈춘 task/run 청소 로직 없음
7. **discovery** — 일부 단지(2942, 3309) `ensure_complex_areas` 가 한 번 실패한 뒤 area 0개로 잔존

## 3. 적용한 수정

### 3.1 마이그레이션 — `20260508_1600_e4f5a6b7c8d9_fix_indexes_and_bigint.py`

```python
# 컬럼 타입 변경
ALTER TABLE listings ALTER COLUMN ask_price TYPE BIGINT

# 중복 transactions 청소 (UNIQUE 적용 전 사전 정리)
DELETE FROM transactions a USING transactions b
 WHERE a.complex_id=b.complex_id AND a.contract_date=b.contract_date
   AND a.price=b.price AND a.exclusive_m2=b.exclusive_m2
   AND COALESCE(a.floor,-1)=COALESCE(b.floor,-1)
   AND a.id < b.id

# 신규 인덱스 (모델에 정의됐으나 DB 미반영)
CREATE UNIQUE INDEX idx_transaction_unique
  ON transactions (complex_id, contract_date, price, exclusive_m2, floor)
CREATE INDEX        idx_transaction_complex_date
  ON transactions (complex_id, contract_date)
CREATE UNIQUE INDEX idx_kb_price_unique
  ON kb_prices (complex_id, area_id, as_of_date)
CREATE INDEX        idx_kb_price_fetched ON kb_prices (fetched_at)
CREATE UNIQUE INDEX ix_listings_source_listing_id ON listings (source_listing_id)
CREATE INDEX        idx_listing_complex_status ON listings (complex_id, status)
CREATE INDEX        idx_listing_fetched ON listings (fetched_at)
```

### 3.2 connector retry — `src/connectors/kb_base.py:_fetch_via_http`

기존 `kb_listing.py` 만 가지고 있던 retry 로직을 base 로 이동.

```python
_TRANSIENT_EXC = (
    httpx.RemoteProtocolError, httpx.ReadTimeout,
    httpx.ConnectTimeout, httpx.ConnectError,
    httpx.WriteError, httpx.PoolTimeout,
)
# 0.5s → 1s → 2s 지수 백오프, 최대 3회
async def _fetch_via_http(self, endpoint, params, max_retries=3):
    for attempt in range(1, max_retries + 1):
        try:
            response = await client.get(...) / client.post(...)
            break
        except self._TRANSIENT_EXC as e:
            if attempt == max_retries: raise NetworkError(...)
            await asyncio.sleep(0.5 * 2**(attempt-1))
```

→ `kb_price`, `kb_transaction`, `kb_facility` 등 모든 KB connector 에 자동 적용.

### 3.3 Transaction UPSERT — `src/workers/tasks.py:collect_kb_transaction_task`

`db.merge()` 는 PK 가 없는 신규 객체에 대해 일반 INSERT 와 동일.
UNIQUE 위반을 정상 처리하려면 PostgreSQL `ON CONFLICT DO NOTHING`.

```python
from sqlalchemy.dialects.postgresql import insert as pg_insert

stmt = pg_insert(Transaction).values(...).on_conflict_do_nothing(
    index_elements=["complex_id", "contract_date", "price",
                    "exclusive_m2", "floor"],
)
res = db.execute(stmt)
if res.rowcount: saved_count += 1
```

### 3.4 Facility UPSERT — `src/workers/tasks.py:ensure_complex_facilities`

KB/OSM 응답에 동일 `external_id` 가 중복 포함되는 케이스가 있어
한 트랜잭션 내에서 INSERT 가 충돌, **트랜잭션 전체가 PendingRollbackError** 가 되어
run_kb_collection 의 task enqueue 까지 모두 실패하던 문제.

```python
stmt = pg_insert(ComplexFacility).values(...).on_conflict_do_nothing(
    index_elements=["complex_id", "facility_type", "external_id"],
)
db.execute(stmt)
```

## 5. 운영 메모

### 새 단지 추가 시 체크리스트
- 단지 등록(discovery) 직후 `ensure_complex_areas` 가 area 를 제대로 생성했는지 확인.
  Area 0개면 transaction/price 수집이 통째로 skip 됨.
- `kb_complex_id` 가 NULL 이면 KB API 조회가 막혀 모든 단지 task 가 no-op.

### Floor NULL 이슈 (잔여)
PostgreSQL UNIQUE 인덱스는 NULL ≠ NULL 로 처리하므로
floor=NULL 인 거래는 idx_transaction_unique 가 잡지 못해 중복이 생길 수 있다.
KB 응답에서 floor NULL 은 매우 드물어 실용적으로 무시 가능.
거슬리면 NULL → 0 정규화 한 줄 추가.

### 모니터링 SQL
```sql
-- 최근 24시간 실패 원인 분포
SELECT error_type, COUNT(*) FROM crawl_tasks
WHERE status='FAILED' AND created_at > now() - interval '24 hours'
GROUP BY error_type ORDER BY 2 DESC;

-- 좀비 후보 (cleanup 가 못 잡은)
SELECT id, run_id, task_key, status, started_at FROM crawl_tasks
WHERE status IN ('RUNNING','PENDING')
  AND started_at < now() - interval '15 minutes';

-- area 0개 단지 (수집 사각지대)
SELECT c.id, c.name FROM complexes c
LEFT JOIN areas a ON a.complex_id = c.id
WHERE c.is_active GROUP BY c.id, c.name HAVING COUNT(a.id) = 0;
```
