### 4-1. Boolean 임포트 누락 (치명적)

**파일**: `src/models/price_data.py`

**증상**:

**원인**: SQLAlchemy `Boolean` 타입을 사용하지만 import 하지 않음
```python
# 수정 전
from sqlalchemy import Column, Integer, String, DateTime, Float, Date, ForeignKey, Text, Index, Enum

# 수정 후
from sqlalchemy import Column, Integer, String, DateTime, Float, Date, ForeignKey, Text, Index, Enum, Boolean
```

**교훈**: SQLAlchemy 모듈이 부분적으로 로드된 후 실패하면, 이미 등록된 테이블 메타데이터 때문에 재임포트 시 별도의 에러가 발생. 근본 원인 추적이 어려움.

## 8. 버그 수정 기록 (추가)

### 8-1. alembic.ini script_location 오류

**파일**: `alembic.ini`

**증상**: Alembic이 env.py를 찾지 못함

**원인**: `script_location = %(here)s` — 프로젝트 루트를 가리킴

**수정**:
```ini
# 수정 전
script_location = %(here)s

# 수정 후
script_location = src/alembic
```

### 8-2. PostgreSQL 포트 충돌

**증상**: psycopg2 UnicodeDecodeError (`'utf-8' codec can't decode byte 0xb8`)

**원인**: 로컬 PostgreSQL(PID 6808)이 포트 5432를 점유하여 Docker 컨테이너 대신 로컬 PostgreSQL에 연결됨. 한국어 Windows(cp949) 환경에서 libpq 에러 메시지 인코딩 불일치 발생.

**진단 과정**:
1. `POSTGRES_HOST_AUTH_METHOD: trust` 시도 → 여전히 실패
2. pg8000 (순수 Python 드라이버) 시도 → 28P01 (password auth failed) 에러
3. `netstat -aon | findstr :5432` → **두 개 프로세스** 발견 (로컬 PG + Docker)
4. wmic으로 PID 확인: 6808=postgres.exe (로컬), 19200=Docker

**수정**:
```yaml
# docker-compose.yml
ports:
  - "5433:5432"    # 5432 → 5433

# .env
DATABASE_URL=postgresql://kb_user:kb_password@localhost:5433/kb_estate
```

**교훈**: 연결 실패 시 반드시 `netstat`으로 포트 점유 프로세스를 확인할 것. 동일 포트에 여러 서비스가 리스닝할 수 있으며, OS가 먼저 바인딩한 프로세스로 연결됨.

## 9. 추가 검증 결과 (2026-02-09 후반)

### 9-1. 매물 API 검증 완료

**문제**: 기존 `stutCdFilter` API는 53개 파라미터 + 특수 헤더가 필요하여 실질적으로 사용 불가

**해결**: 새로운 2단계 API 발견 및 검증
1. `GET /land-complex/complex/brif` — 단지 브리프 정보 (매매/전세/월세 건수 포함)
2. `POST /land-property/propList/main` — 매물 상세 목록 (brif 데이터를 POST body로 전송)

**핵심 발견**: propList/main은 **HTTP/2 필수** — HTTP/1.1 사용 시 `RemoteProtocolError: Server disconnected`

**검증 결과**: 794개 단지에서 **7,053개 매물 수집** 성공 (에러 1건, 연결 끊김)

### 9-2. 실거래가 데이터 소스 확인

**문제**: `vlaDealDtailPriceInq` API가 모든 단지에 대해 "No Data" 반환

**해결**: `BasePrcInfoNew` 응답 내 `최근실거래가` 필드에서 실거래 데이터 추출 → 별도 API 호출 불필요

### 9-3. 가격 데이터 오버플로 수정

**문제**: Integer 컬럼 (max ~21.4억)으로 고가 아파트(28억+) 가격 저장 실패

**수정**: `kb_prices`, `transactions`, `listings` 테이블의 모든 가격 컬럼을 `BigInteger`로 변경
