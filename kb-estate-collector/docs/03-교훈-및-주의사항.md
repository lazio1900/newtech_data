# 교훈 및 주의사항

## 1. kbland.kr은 SPA — 정적 크롤링 불가

**문제**: kbland.kr은 Vue.js 기반 SPA로, HTML을 가져오면 빈 껍데기만 옵니다.
- `requests.get()`, `WebFetch`, `httpx.get()` 모두 실제 데이터를 받아올 수 없음
- JavaScript가 실행되어야 내부 API(`data-api.kbland.kr`, `api.kbland.kr`)를 XHR로 호출

**해결**: Playwright로 브라우저를 렌더링한 후 네트워크 요청을 인터셉트하는 방식 채택

**교훈**: 한국 금융/부동산 사이트 대부분이 SPA 구조. 크롤링 전 반드시 SPA 여부를 확인할 것.

## 2. KB부동산은 공개 API를 제공하지 않는다

**문제**: 공식 API 문서, 개발자 포털 없음. PublicDataReader 라이브러리가 있지만 통계 데이터(가격지수, 평균가격)만 제공하며, 개별 단지의 시세/실거래가/매물 데이터는 제공하지 않음.

**해결**:
- 내부 API 자동 발견 도구(`api_discovery.py`) 개발
- 하이브리드 접근: 발견된 API를 httpx로 직접 호출 시도 → 실패 시 Playwright 폴백

**교훈**: 공개 API가 없는 사이트는 "API 디스커버리" 단계를 설계에 포함시킬 것.

## 3. API 응답 구조를 사전에 알 수 없다

**문제**: API 디스커버리 전에는 정확한 응답 필드명을 알 수 없음. `dealAmt`인지 `거래금액`인지 `price`인지...

**해결**: `parse()` 메서드에 "후보 키 리스트" 패턴 적용
```python
# 여러 후보 키에서 값 추출
for key in ["dealAmt", "general_price", "avgPrc", "매매가"]:
    if key in data:
        return self._to_won(data[key])
```

**교훈**: 역공학 기반 크롤러에서는 방어적 파싱이 필수. 하나의 키만 기대하면 사이트 변경 시 바로 깨짐.

## 4. async/sync 경계 문제

**문제**: Playwright는 async, Celery 태스크는 sync. 둘을 연결하려면 이벤트 루프 관리가 필요.

**해결**:
```python
def run_async(coro):
    loop = asyncio.new_event_loop()
    try:
        return loop.run_until_complete(coro)
    finally:
        loop.close()
```

`KBBaseConnector.fetch()`에서도 `asyncio.run()` 또는 ThreadPoolExecutor를 통해 async→sync 브릿지 처리.

**교훈**: 이벤트 루프 중첩(`asyncio.run()` inside running loop)을 주의할 것. 각 태스크마다 새 루프를 생성하는 것이 안전.

## 5. 안티 디텍션은 다층 방어

**적용된 대책**:
1. **User-Agent 풀**: 7개의 실제 Chrome UA 중 랜덤 선택
2. **navigator.webdriver 오버라이드**: headless 탐지 방지
3. **navigator.plugins/languages 위장**: 빈 값 탐지 방지
4. **랜덤 딜레이**: 2~5초 인간 행동 모방
5. **레이트 리밋**: 분당 20회 요청 제한 (보수적)
6. **한국어 로케일/서울 타임존**: 자연스러운 브라우저 환경

**교훈**: 하나의 대책만으로는 부족. 여러 계층의 위장이 필요.

## 6. 매물 데이터의 상태 관리

**문제**: 매물은 등장/소멸이 잦음. "이번에 안 보인 매물"을 어떻게 처리할 것인가?

**해결**: `collect_kb_listing_task`에서:
1. 이번 수집에서 발견된 매물 ID를 `seen_ids`에 수집
2. 기존 ACTIVE 매물 중 `seen_ids`에 없는 것 → `REMOVED`로 상태 변경
3. 기존 매물이 다시 보이면 → `ACTIVE`로 복원, 가격 업데이트

**교훈**: 매물 데이터는 "없음"도 정보. 상태 변경 추적이 분석에 매우 중요.

## 7. Windows 환경 주의사항

- `ls` 명령어 사용 불가 → `dir` 또는 Glob 도구 사용
- 파일 경로에 `\` 백슬래시 사용
- Playwright 설치 시 `playwright install chromium`이 별도 필요

## 8. 구현 순서의 중요성

가장 효율적인 구현 순서:
```
의존성 → 인프라(브라우저) → 디스커버리 → 커넥터 → 서비스 → 태스크 통합
```

**핵심 게이트**: API 디스커버리(`scripts/run_api_discovery.py`)를 실행하여 실제 엔드포인트를 확인한 후에야 커넥터의 `parse()` 메서드를 정확하게 구현할 수 있음.

## 9. 기존 아키텍처 활용

이미 잘 설계된 구조가 있었기 때문에 효율적으로 개발 가능:
- `BaseConnector`의 재시도/레이트리밋 로직 재활용
- `DatabaseTask`의 세션 관리 패턴 그대로 사용
- `CrawlRun/CrawlTask` 추적 모델 그대로 사용

**교훈**: 기존 추상화 계층이 좋으면 확장이 쉽다. `BaseConnector` → `KBBaseConnector` 상속만으로 HTTP/Browser 하이브리드 전략을 깔끔하게 추가할 수 있었음.

## 10. `db.merge()` ≠ UPSERT (5월 추가)

**문제**: 5월에 `idx_transaction_unique` UNIQUE 인덱스 추가 후 IntegrityError 폭발.
원인은 `Transaction` 저장에 `db.merge()` 사용 — PK 가 없는 신규 객체에 merge 는 일반 INSERT 와 동일하므로 UNIQUE 위반을 막지 못함.

**해결**:
```python
from sqlalchemy.dialects.postgresql import insert as pg_insert
stmt = pg_insert(Model).values(...).on_conflict_do_nothing(
    index_elements=["complex_id", "contract_date", "price", ...]
)
```

**교훈**: SQLAlchemy `merge()` 는 PK 비교 기반. 비-PK UNIQUE 제약 위에서 UPSERT 가 필요하면 dialect-specific `on_conflict_do_*()` 를 직접 써야 한다. `Transaction` / `ComplexFacility` 모두 같은 패턴.

## 11. PostgreSQL UNIQUE 인덱스의 NULL 의미 (5월 추가)

PostgreSQL UNIQUE 는 `NULL ≠ NULL` 로 판정 — 즉 컬럼이 NULL 인 row 는 중복 검사를 통과한다.
`idx_transaction_unique (..., floor)` 에서 floor=NULL 거래는 중복 INSERT 가 그대로 들어간다.

**대응**: 마이그레이션에서 중복 정리할 때는 `COALESCE(floor,-1)` 로 NULL 동등 비교.
실제 KB 응답에서 floor NULL 은 매우 드물어 운영상 무시. 거슬리면 NULL → 0 정규화.

## 12. macOS prefork + 네이티브 라이브러리 fork-safety (5월 추가)

**문제**: prefork 풀로 worker 기동 시 SIGSEGV 가 빈발 → 좀비 task 누적 → 단지 수집이 통째로 멈춰 보임.

**원인**: macOS Objective-C 런타임이 fork 후 자식 프로세스에서 초기화될 때 안전성을 검증하는데, 일부 라이브러리(특히 httpx + h2 + native bindings)와 충돌.

**해결**:
```bash
ENV: OBJC_DISABLE_INITIALIZE_FORK_SAFETY=YES
ENV: PYTHONFAULTHANDLER=1
celery worker --pool=prefork --concurrency=4
```

threads pool 은 `DatabaseTask._db` 가 클래스 변수라 race 위험 → prefork 유지.

## 13. 좀비 task/run — 워커 종료 추적 (5월 추가)

worker 가 SIGSEGV/OOM/Ctrl+C 로 종료되면 진행 중이던 task/run 은 status=RUNNING 인 채로 DB 에 남는다. 이게 누적되면 `_finalize_run_if_complete` 가 영원히 안 끝나 통계가 깨지고, 새 run 도 영향받는다.

**대응**: `cleanup_zombie_runs` task — 5분 주기로 cutoff 초과 RUNNING/PENDING 을 FAILED 로 마감.
- task: 10분 timeout
- run: 60분 timeout (대규모 수집 고려)

**교훈**: 분산 task 시스템에서는 "비정상 종료 잔재"를 청소하는 책임을 코드에 명시적으로 넣어야 한다. broker/worker heartbeat 만으로는 DB 쪽 status 가 자동 보정되지 않는다.

## 14. RedBeat — 동적 cron 의 운영 형태 (5월 추가)

CeleryBeat 의 기본 PersistentScheduler 는 schedule 변경 시 beat 재기동이 필요. 다중 단지/지역 단위 cron 을 사용자가 UI 에서 자주 조정하려면 `redbeat.RedBeatScheduler` 가 적합.

```python
celery_app.conf.beat_scheduler = "redbeat.RedBeatScheduler"
celery_app.conf.redbeat_redis_url = settings.celery_broker_url
celery_app.conf.beat_max_loop_interval = 15  # 15초마다 redis 폴링
```

DB 의 `crawl_jobs.cron_schedule` 변경 시 `_sync_redbeat_entry()` 호출 → Redis 즉시 반영 → 다음 폴링 cycle 에서 적용.

정적 task (예: `cleanup_zombie_runs`) 는 `celery_app.conf.beat_schedule` 에 두면 RedBeat 도 자동 픽업.

**교훈**: schedule 이 사용자 입력에 의해 자주 바뀌는 시스템이면 Beat 재기동 비용을 처음부터 설계에서 제거할 것.
