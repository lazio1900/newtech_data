# 프론트엔드 및 전체 기동 가이드

## 개요

2026-02-09 기준으로 프론트엔드 웹 UI와 백엔드 인프라가 모두 구축되어,
브라우저에서 크롤링을 작동시키고 수집된 데이터를 확인할 수 있는 상태입니다.

## 3. 화면 구성

### 3-1. 대시보드 (`/`)
- 등록 단지 수, 활성 작업 수, 최근 실행 수, 수집 데이터 건수 카드
- 최근 실행 목록 (상태 뱃지, 성공/실패 건수)
- 실행 통계 바 차트

### 3-2. 단지 관리 (`/complexes`)
- 단지 목록 (이름, 주소, 우선순위, 면적 수)
- 검색 필터
- **"지역 발견"** 버튼 → 서울 구 코드 입력 → KB API로 해당 지역 아파트 단지 자동 등록
- 단지 상세 → 면적 정보, 시세 차트, 실거래가, 매물 탭

### 3-3. 수집 작업 (`/jobs`)
- 작업 목록 (유형, 상태, 스케줄)
- 실행/일시정지/재개 버튼
- **"지역 수집"** 버튼 → 지역 코드 선택 → 발견→시세→실거래→매물 한번에 수집

### 3-4. 실행 이력 (`/runs`)
- 실행 목록 (상태, 시작/종료, 소요시간, 성공/실패 건수)
- 실행 상세 → 태스크 목록 (상태 필터링 가능)

### 3-5. 데이터 탐색 (`/data`)
- 단지 선택 드롭다운
- **KB 시세** 탭: 시세 추이 라인차트 + 데이터 테이블 + CSV 내보내기
- **실거래가** 탭: 계약일, 거래가, 면적, 층 테이블
- **매물** 탭: 호가, 면적, 층, 상태 테이블 (ACTIVE/REMOVED 필터)

## 4. API 프록시 설정

Vite dev server가 `/api` 요청을 FastAPI 백엔드로 프록시합니다:

```typescript
// vite.config.ts
server: {
  port: 5173,
  proxy: {
    "/api": {
      target: "http://localhost:8000",
      changeOrigin: true,
    },
  },
}
```

- 프론트엔드: `http://localhost:5173` (또는 5174)
- API 요청: `/api/complexes/` → `http://localhost:8000/api/complexes/`

## 5. 전체 시스템 기동 순서

### 사전 요구사항
- Python 3.11+
- Node.js 22.12+ (Vite 7 요구)
- Docker Desktop (PostgreSQL + Redis용)

### Step 1: Docker 인프라

```bash
cd kb-estate-collector
docker-compose up -d postgres redis
```

> **주의**: 로컬에 PostgreSQL이 설치되어 있으면 5432 포트 충돌이 발생합니다.
> `docker-compose.yml`에서 포트를 `5433:5432`로 매핑하고, `.env`의 `DATABASE_URL`도 수정하세요.

확인:
```bash
docker-compose ps
# postgres (healthy), redis (healthy) 확인
```

### Step 2: DB 마이그레이션

```bash
cd kb-estate-collector

# 마이그레이션 파일 생성 (최초 1회)
python -m alembic revision --autogenerate -m "Initial schema"

# 마이그레이션 적용
python -m alembic upgrade head
```

생성되는 테이블 (10개):
```
alembic_version, complexes, areas, crawl_jobs, crawl_runs,
crawl_tasks, kb_prices, transactions, listings, raw_payloads
```

### Step 3: FastAPI 백엔드

```bash
cd kb-estate-collector
uvicorn src.admin_api.main:app --host 0.0.0.0 --port 8000 --reload
```

확인: http://localhost:8000/docs (Swagger UI)

### Step 4: Celery Worker

```bash
# 별도 터미널
cd kb-estate-collector
celery -A src.workers.celery_app worker --loglevel=info --pool=solo
```

> **Windows**: `--pool=solo` 필수 (fork 미지원)

등록 태스크 8개:
```
src.workers.tasks.collect_kb_price_task
src.workers.tasks.collect_kb_transaction_task
src.workers.tasks.collect_kb_listing_task
src.workers.tasks.discover_complexes_task
src.workers.tasks.run_kb_price_collection
src.workers.tasks.run_transaction_collection
src.workers.tasks.run_listing_collection
src.workers.tasks.run_region_collection
```

### Step 5: Frontend

```bash
# 별도 터미널
cd kb-estate-collector/frontend
npm install    # 최초 1회
npm run dev
```

확인: http://localhost:5173 (또는 5174)

## 7. 사용 시나리오

### 시나리오 1: 강남구 아파트 전체 수집

1. 브라우저에서 http://localhost:5173 접속
2. **단지 관리** 메뉴 클릭
3. **"지역 발견"** 버튼 클릭
4. 서울 강남구 (`11680`) 선택 또는 입력
5. → KB API가 강남구 좌표 범위 내 아파트 단지를 자동 검색하여 DB에 등록
6. **수집 작업** 메뉴 → **"지역 수집"** 버튼 클릭
7. 지역 코드 `11680` 입력, 수집 유형 선택 (시세/실거래가/매물)
8. → Celery가 백그라운드에서 각 단지별 데이터 수집 시작
9. **실행 이력** 메뉴에서 진행 상태 실시간 확인
10. **데이터 탐색** 메뉴에서 단지별 시세 차트/실거래가/매물 확인

### 시나리오 2: 특정 단지 시세 확인

1. **단지 관리** → 단지 목록에서 원하는 단지 클릭
2. 단지 상세 → 면적 정보 확인
3. 시세 탭: 시세 추이 차트 + 데이터 테이블
4. CSV 내보내기 버튼으로 엑셀 분석용 데이터 다운로드

### 시나리오 3: CLI로 직접 수집 실행

```bash
# 강남구 단지 발견
curl -X POST "http://localhost:8000/api/complexes/discover-region?region_code=11680"

# 지역 전체 수집 (발견→시세→실거래가→매물)
curl -X POST "http://localhost:8000/api/jobs/run-region?region_code=11680"

# 수집된 시세 확인
curl "http://localhost:8000/api/data/kb-prices?complex_id=1"

# CSV 내보내기
curl "http://localhost:8000/api/data/kb-prices/export?complex_id=1" > prices.csv
```

## 8. 트러블슈팅

### PostgreSQL 연결 실패 (psycopg2 UnicodeDecodeError)

```
UnicodeDecodeError: 'utf-8' codec can't decode byte 0xb8 in position 63
```

**원인**: 로컬에 PostgreSQL이 설치되어 있어 포트 5432 충돌. Python이 Docker 컨테이너가 아닌 로컬 PostgreSQL에 연결되었고, 한국어 Windows(cp949)에서 libpq 에러 메시지 인코딩 불일치 발생.

**해결**:
1. `docker-compose.yml`에서 PostgreSQL 포트를 `5433:5432`로 변경
2. `.env`의 `DATABASE_URL`을 `postgresql://kb_user:kb_password@localhost:5433/kb_estate`로 변경
3. `docker-compose down -v && docker-compose up -d postgres redis`

### Alembic versions 디렉토리 없음

```
FileNotFoundError: src/alembic/versions/...
```

**해결**: `mkdir src/alembic/versions` 후 재실행

### Alembic script_location 오류

```
FAILED: No such file or directory: '%(here)s/env.py'
```

**해결**: `alembic.ini`의 `script_location`을 `src/alembic`으로 수정

### Vite 포트 충돌

```
Port 5173 is in use, trying another one...
```

**해결**: 자동으로 5174 등 다음 포트 사용. 정상 동작에 영향 없음.

### Celery Windows 오류

```
ValueError: not enough values to unpack (expected 3, got 0)
```

**해결**: `--pool=solo` 옵션 추가. Windows에서는 fork가 지원되지 않아 solo pool 필수.

## 9. 환경 설정 파일

### .env
```env
# Database (port 5433 - 로컬 PG 충돌 방지)
DATABASE_URL=postgresql://kb_user:kb_password@localhost:5433/kb_estate

# Redis
REDIS_URL=redis://localhost:6379/0

# Celery
CELERY_BROKER_URL=redis://localhost:6379/0
CELERY_RESULT_BACKEND=redis://localhost:6379/0

# API
API_HOST=0.0.0.0
API_PORT=8000

# Logging
LOG_LEVEL=INFO
```

### docker-compose.yml 핵심 설정
```yaml
services:
  postgres:
    image: postgres:14-alpine
    ports:
      - "5433:5432"     # 로컬 PG 충돌 방지
    environment:
      POSTGRES_USER: kb_user
      POSTGRES_PASSWORD: kb_password
      POSTGRES_DB: kb_estate

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
```
